# /finding-drafting

## 重要：先加载 Practice Profile

读取 → 审计发现分类标准：重大缺陷/重要发现/改善建议/观察点的定义。

## 工作流

### Step 1：确认发现的事实基础

在起草前，确认以下问题已明确：
- 已获得充分证据支持该发现？*（证据清单已整理？）*
- 已与被审计单位初步沟通？*（给了解释机会？）*
- 已排除已知的例外或特殊情形？

### Step 2：五要素结构

每个审计发现严格按照五要素编写：

```
┌─ 审计发现 [编号] ────────────────────────────────────────────┐
│                                                               │
│ 🔴 / 🟠 / 🟡 / 🟢   [分类等级]                                 │
│                                                               │
│ 条件（Condition）：                                            │
│ 描述实际观察到的情况，包括时间、地点、涉及金额或数量。               │
│ "我们在 2024 年 12 月的测试中发现，25 笔采购订单中有 3 笔         │
│  缺失二级审批签字，涉及金额 ¥156 万元。"                         │
│                                                               │
│ 标准（Criteria）：                                              │
│ 指出应该遵循的制度、流程、法规。                                  │
│ "根据公司《采购管理制度》第 5.3 条，单笔金额超过 ¥50 万元的       │
│  采购订单须经部门总监和采购经理双签审批。"                        │
│                                                               │
│ 原因（Root Cause）：                                            │
│ 分析问题的根本原因。                                             │
│ "经了解，造成该问题的主要原因是：1) 审批人出差时未设置代理审批人； │
│  2) 系统未在审批人缺席时自动升级审批流程。"                      │
│                                                               │
│ 影响（Impact）：                                                │
│ 描述问题带来的实际或潜在影响。                                    │
│ "可能导致未经充分审批的采购行为，增加公司资金风险和合规风险。       │
│  未及时纠正可能导致损失扩大。"                                   │
│                                                               │
│ 建议（Recommendation）：                                         │
│ 提出可操作的改进建议。                                           │
│ "1. 建立审批人出差代理机制，关键岗位须设置 A/B 角                │
│  2. 优化系统审批流：审批人超 24 小时未处理自动升级至上一级        │
│  3. 对当前缺失审批的 3 笔订单补充追认审批"                        │
└─────────────────────────────────────────────────────────────────┘
```

### Step 3：分类定级

根据 Practice Profile 的分类标准判断等级：

```
┌─ 发现定级 ────────────────────────────────────────────┐
│ 因素               │ 判断结果                        │
│ 影响程度           │ 可能产生财务损失 ¥50 万以上      │
│ 发生可能性         │ 抽样发现 3/25 = 12% 偏差率      │
│ 涉及合规法规       │ 不直接涉及                      │
│ 控制缺失类型       │ 控制存在但执行不到位             │
│                     │                                │
│ 定级结论           │ 🟠 重要发现                     │
└─────────────────────────────────────────────────────┘
```

### Step 4：证据索引

每个发现须引用支撑证据：

```
证据清单：
[E-001] 采购订单 PO-2024-0856 ~ PO-2024-0875 签字扫描件（25 份）
[E-002] 审批人张三 2024 年 12 月出差记录
[E-003] 采购管理制度（v3.2）第 5.3 条
[E-004] 与被审计单位沟通记录（2025-01-10 会议纪要）
```

### Step 5：与被审计单位沟通

完成初稿后 → 发送审计发现确认函：

```
┌─ 审计发现确认函 ─────────────────────────────────────────────┐
│ 致：[被审计单位负责人]                                          │
│                                                               │
│ 我们在本次审计中识别到以下发现，现征求贵方意见。                    │
│ 请于 [日期] 前书面回复确认或提出异议。                            │
│                                                               │
│ 发现 [编号]：[标题]                                             │
│ 事实描述：[引用条件部分]                                         │
│ 初步定级：🟠 重要发现                                            │
│                                                               │
│ 贵方回复：                                                      │
│ □ 确认事实无误                                                  │
│ □ 部分有异议（说明：__）                                         │
│ □ 不同意（理由：__）                                             │
│ ─────────────────────────                                      │
│ 被审计单位签字：________ 日期：________                         │
└─────────────────────────────────────────────────────────────────┘
```

### Step 6：输出

```
[项目编号]_FD_[日期].md — 审计发现清单（征求意见稿）
├── 发现 1：[标题]（等级：🔴/🟠/🟡/🟢）
├── 发现 2：[标题]（等级：🔴/🟠/🟡/🟢）
├── ...
├── 发现 N：[标题]（等级：🔴/🟠/🟡/🟢）
├── 证据索引
└── 分类统计：🔴 N 项 / 🟠 N 项 / 🟡 N 项 / 🟢 N 项
```
