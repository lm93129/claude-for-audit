# /wp-generator

## 重要：先加载 Practice Profile

读取 `~/.claude/plugins/config/claude-for-audit/statutory-audit/CLAUDE.md` → 底稿格式：
- 底稿编码规则
- 底稿语言（中文/英文/双语）
- 纸质/电子存档
- 使用审计软件

## 适用场景

这个技能在以下情况下使用：
1. 各审计程序执行完毕 → 统一整理为结构化底稿
2. 底稿归档前 → 检查完整性和格式
3. 复核过程中 → 为复核人提供可读的底稿包
4. 多份工作成果需合并 → 汇总到一个底稿文件中

## 工作流

### Step 1：选择底稿类型

| 类型 | 底稿编码前缀 | 包含内容 |
|------|------------|---------|
| 综合类 | A | 审计计划、风险评估、重要性、审计报告 |
| 收入循环 | B1 | 收入测试、应收账款函证、收入截止测试 |
| 采购循环 | B2 | 采购测试、应付账款函证 |
| 生产循环 | B3 | 存货监盘、计价测试 |
| 薪酬循环 | B4 | 工资测试、社保计提 |
| 投资筹资循环 | B5 | 借款测试、权益测试 |
| 货币资金 | B6 | 银行函证、余额调节表 |
| 其他类 | C | 关联方、期后事项、持续经营评估 |

用户选择类型或系统根据已有程序执行记录自动判断。

### Step 2：收集各程序的执行成果

自动检测已有底稿文件或接收用户上传：

```
检测到以下程序已执行：
✓ risk-assessment      → [底稿编码]-RA-01 ~ 04
✓ materiality          → [底稿编码]-MAT-01
✓ analytical-review     → [底稿编码]-AR-01
✓ confirmations        → [底稿编码]-CNF-01 ~ 03
✓ substantive-testing   → [底稿编码]-ST-01 ~ 04

未执行的程序（可选追加）：
○ ic-review
○ sampling
○ journal-entry-testing
```

### Step 3：生成单张底稿

**示例——底稿格式：**

```
┌─────────────────────────────────────────────────────────┐
│ [事务所名称]                                               │
│ ─────────────────────────────────────────────────        │
│ 审计工作底稿                                               │
│                                                          │
│ 客户：[被审计单位名称]   报表日：2024年12月31日               │
│ 底稿编号：[底稿编码规则]                                     │
│ 科目：[科目名称]         币种：人民币                       │
│ ─────────────────────────────────────────────────        │
│                                                          │
│ 索引目录：                                                  │
│ 一、审计目标                                               │
│ 二、审计程序 ..................................................................  │
│ 三、审计发现 ..................................................................  │
│ 四、审计结论 ..................................................................  │
│ 五、复核记录                                             │
│                                                          │
│ ─────────────────────────────────────────────────        │
│ 编制人：[姓名]   编制日期：[日期]   复核人：[姓名]   复核日期：[日期] │
│ 复核状态：[未复核/一级完成/二级完成/三级完成]                    │
└─────────────────────────────────────────────────────────┘
```

### Step 4：生成底稿包

生成包含以下内容的底稿包：

```
[底稿编码]-WP-INDEX 底稿目录.xlsx
  → Sheet 1：底稿索引（所有底稿编号、名称、编制人、复核状态）
  → Sheet 2：未完成事项清单
  → Sheet 3：复核汇总

[底稿编码]-WP-[科目] 各科目底稿.xlsx
  → 独立底稿文件，按编码命名
```

### Step 5：完整性检查

```
┌─ 底稿完整性检查 ───────────────────────────────────────┐
│ 项目               │ 状态 │ 备注                       │
│ 底稿头（事务所信息）   │ ✓   │                           │
│ 客户名称和报表日     │ ✓   │                           │
│ 底稿编码             │ ✓   │ [编码]                    │
│ 编制人签名和日期     │ ✓   │ [姓名/日期]                │
│ 审计程序索引         │ ✓   │ 已列明所有执行程序            │
│ 审计证据索引         │ ⚠   │ 确认函扫描件待上传            │
│ 复核记录            │ ——  │ 一/二/三级尚未完成            │
│ 结论部分            │ ✓   │                           │
└──────────────────────────────────────────────────────────┘
```

### Step 6：输出

```
[底稿编码]-WP-INDEX 底稿目录.xlsx
[底稿编码]-WP-[科目1].xlsx
[底稿编码]-WP-[科目2].xlsx
...

⚠️ 底稿包仅限项目组内部使用。
⚠️ 底稿归档要求：审计报告日后 60 天内完成最终归档。
⚠️ 所有电子底稿应保证不可修改格式（建议输出为 PDF 存档版本）。
```
