# 示例案例：APP 用户信息处理个保法专项合规审计

## 场景简介

被审计对象：某消费类 APP 产品团队（虚构）
主题：用户信息收集合规专项
目标：演示 `compliance-audit` 如何完成一条从法规适用识别到个保法检查的问题闭环。

## 场景背景（示例）

- APP 提供会员服务、内容推荐、商城下单和客服功能
- 采集信息包括：手机号、昵称、头像、收货地址、设备标识、定位信息、浏览行为、订单记录
- 与第三方共享对象包括：云服务商、短信服务商、支付渠道、广告投放平台、客服外包
- 已发布隐私政策，但最近一年未系统更新
- 产品新增了“猜你喜欢”和“附近门店”功能

## 演示路径

### 标准路径

1. `/compliance-audit:cold-start-interview`
2. `/compliance-audit:regulatory-checker`
3. `/compliance-audit:pip-compliance`
4. 如涉及外包客服/广告投放/代理渠道 → `/compliance-audit:anti-corruption`（可选，检查第三方管理）
5. 输出整改清单并纳入跟踪

## 示例输入信息

### 1. 处理活动清单（简化版）

| 场景 | 处理目的 | 数据类型 | 敏感性 |
|------|---------|---------|-------|
| 用户注册 | 创建账号 | 手机号、验证码 | 一般 |
| 下单配送 | 履行订单 | 姓名、手机号、地址 | 一般 |
| 个性化推荐 | 提升转化 | 浏览行为、点击、停留时长 | 一般 |
| 附近门店 | 推荐门店 | 精确定位 | 敏感 / 高关注 |
| 客服工单 | 处理投诉 | 账号信息、订单信息、聊天记录 | 一般 |
| 画像营销 | 精准营销 | 用户行为标签 | 高关注 |

### 2. 已知问题线索

- 隐私政策最近更新时间为 14 个月前
- “附近门店”功能收集精确定位，但未见单独同意弹窗证据
- 个性化推荐关闭入口不明显
- 第三方共享清单未列出广告投放平台的最新供应商
- 用户删除账号后，后台留存时间规则不清晰
- 未找到针对敏感个人信息处理的专项风险评估记录

### 3. 可供法规适用识别的场景描述

- APP 面向中国大陆个人用户
- 通过手机号注册
- 使用个性化推荐、营销推送和位置服务
- 与支付、短信、云服务和广告平台共享部分信息
- 没有明确的跨境传输说明

## 预期演示输出

### 1. `regulatory-checker`

预期识别的法规层级：
- 《个人信息保护法》
- 《数据安全法》
- 《网络安全法》
- 与 APP/隐私政策/推荐算法/第三方共享相关的监管要求
- 公司内部的隐私政策、用户协议、第三方管理制度

输出重点：
- 合法性基础
- 最小必要
- 单独同意
- 第三方共享透明度
- 个人权利响应
- 留存期限

### 2. `pip-compliance`

大概率会识别的高风险问题：
- ⚠ 精确定位的单独同意机制不足
- ⚠ 个性化推荐关闭/拒绝路径不清晰
- ⚠ 第三方共享对象清单不完整
- ⚠ 账号删除后的留存规则不明确
- ⚠ 敏感个人信息或高风险处理活动缺少专项评估记录

### 3. 可能的整改建议方向

1. 对定位信息处理补充单独同意和显著提示
2. 优化推荐关闭入口和撤回同意机制
3. 更新隐私政策和第三方共享清单
4. 建立个人信息处理活动台账
5. 明确保留期限与删除/匿名化规则
6. 对高风险处理活动补做专项评估

## 推荐扩展

做完这个示例后，可继续扩展：
- 员工信息处理合规专项
- 数据出境专项检查
- 广告推荐和画像营销专项
