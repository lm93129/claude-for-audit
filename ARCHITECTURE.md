# 架构设计文档

本仓库的设计融合了 `anthropics/financial-services` 和 `anthropics/claude-for-legal` 两个官方参考仓库的最佳实践。

## 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                    Practice Profile                      │
│  ~/.claude/plugins/config/claude-for-audit/<plugin>/     │
│                    CLAUDE.md                              │
│  ┌────────────┬────────────┬──────────┬────────────────┐ │
│  │ 事务所信息   │ 方法论偏好  │ 连接器状态│ 独立性配置       │ │
│  └────────────┴────────────┴──────────┴────────────────┘ │
└─────────────────────┬───────────────────────────────────┘
                      │ 所有技能从这里读取配置
                      ▼
┌─────────────────────────────────────────────────────────┐
│                Skills（审计技能）                          │
│                                                          │
│  每个技能 = SKILL.md                                     │
│  ├── Frontmatter：触发条件                                 │
│  ├── 先读 Practice Profile                                │
│  ├── 执行审计程序                                          │
│  ├── 输出底稿/发现/报告草稿                                  │
│  └── 标注 ⚠️ 证据链来源 + 复核要求                           │
└─────────────────────────────────────────────────────────┘
                      │ 定时任务和监控
                      ▼
┌─────────────────────────────────────────────────────────┐
│                Agents（定时代理）                           │
│                                                          │
│  deadline-watcher: 截止日跟踪                              │
│  continuous-monitor: 持续审计监控                           │
│  anomaly-detector: 异常交易预警                             │
└─────────────────────────────────────────────────────────┘
```

## 插件生命周期

```
① 安装
   │
   ▼
② 冷启动采访（客户必做，不可跳过）
   │  └── 写入 Practice Profile + 公司级 profile
   ▼
③ 日常使用
   │  └── 技能自动触发 / 用户手动触发
   ▼
④ 局部调整（customize 技能）
   │  └── 改一个参数，不用重跑采访
   ▼
⑤ 方法论文档变更 → 重跑采访 --redo
```

## 数据流

```
数据源 ──MCP──┐
              ├──► 技能执行 ──► 工作底稿（.xlsx / .md）
审计人员输入 ──┘                │
                               ├──► 审计发现清单
                               ├──► 复核意见
                               └──► 审计报告草稿
```

### 验证管理层要求

每个输出必须满足"3C"标准：

| 标准 | 要求 | 示例 |
|------|------|------|
| **来源（Cite）** | 每个数字的来源出处 | "凭证号 P-2024-001，科目余额表第 3 行" |
| **方法（Criteria）** | 引用对应的审计准则 | "根据 CSA 1311 号，函证未回函需执行替代程序" |
| **结论前需复核（Check）** | 标注复核状态 | "【待项目组复核】【底稿编码 A1-01-001】" |

## 安全与职业道德护栏

### 1. 独立性门控

每个审计程序执行前，技能自动检查：
- 被审计单位是否在本期受禁服务清单中
- 签字CPA是否处于该客户的冷却期
- 当前程序是否属于非鉴证服务的禁止范围

### 2. 角色门控

```
执业CPA          → 可输出审计结论草稿、签字意见
审计助理          → 只能输出"程序执行记录"，不能给出结论
非审计人员        → 不能接触底稿和调整分录
```

### 3. 输出分类

每个输出自带 Tier 标注：

| Tier | 标签 | 谁来用 | 可否对外 |
|------|------|--------|---------|
| Tier 1 工作底稿 | ⚠️ 底稿 | 项目组内部 | 否 |
| Tier 2 审计发现 | 🔍 发现 | 与被审计单位沟通 | 需复核 |
| Tier 3 审计报告 | 📄 报告 | 对外发布 | 需签字 |

### 4. 证据链护栏

```
┌─ 每个数字 ─┐
│ 来源：凭证号/对账单编号/系统截图     │
│ 方式：检查/观察/询问/函证/重新计算   │
│ 索引：对应工作底稿编码               │
│ 复核：未复核 / 一级复核 / 二级复核    │
└─────────────────────────────────────┘
```

## MCP 数据连接器设计

推荐使用的中国审计场景连接器：

| 连接器 | 用途 | 认证方式 |
|--------|------|---------|
| 用友/金蝶 ERP | 获取会计数据 | API Key |
| 银行接口 | 自动获取银行对账单 | 数字证书 |
| 企查查/天眼查 | 工商信息、关联方 | API Token |
| 电子税务局 | 纳税申报数据 | 企业账号 |
| 企业微信/飞书/钉钉 | 沟通与文件传输 | OAuth |
| 内部审计系统 | 审计底稿管理系统 | SSO |

## 与中国审计准则的对齐

每个技能在适用时引用对应的 CSA 准则编号：

| 技能 | 对应 CSA | 内容 |
|------|---------|------|
| risk-assessment | CSA 1211 第 × 条 | 了解被审计单位及其环境 |
| materiality | CSA 1221 第 × 条 | 计划和执行审计工作的重要性 |
| ic-review | CSA 1231 第 × 条 | 针对评估的重大错报风险采取的应对措施 |
| sampling | CSA 1310 第 × 条 | 审计抽样 |
| confirmations | CSA 1312 第 × 条 | 函证 |
| analytical-review | CSA 1313 第 × 条 | 分析程序 |
| wp-generator | CSA 1201 第 × 条 | 审计工作底稿
